# 34. Programmatic SEO とデータ — KBから数万ページ規模への展開設計

> V2 25章(114案・シンコンテンツゲート)のデータ面の裏付けを定義する。**V2 25章のゲートはV3でも全面的に有効** — 本章は「何ページ作れるか(能力)」を設計するのであって「何ページ公開するか(判断)」を変えない。

## 1. 基本式 — ページ=テンプレート × エンティティ結合

```
ページ群 = 検索意図テンプレート(T) × KBデータセットの結合(J) × 公開ゲート(G)
例: #1 都道府県×介護職の給料
    T: 「(県名)の介護職の給料」の意図に応えるT5系テンプレ
    J: A4 prefectures ⋈ C2 salary-stats(prefSlug, 最新year)⋈ A2 jobs
    G: 条件0(実在確認済) + null率<30% + 独自ブロック400字 + …(V2 25章 §4)
```

- V2 25章の**データ駆動系(A〜E群・G群・ツール)は J(結合式)として書き直せる**。F群(悩み・転職=状況軸)と #14(月給レンジ軸)は記事系であり、KBは素材参照に留まる(軸エンティティを31章に持たない — パネル A-05)。25章が「案の台帳」、本章が「その生成機構」。
- **結合式 J はコードではなくデータとして持つ**: E7 pSEOページ群レジストリ(31章 §2 — templateId・URLパターン・参照データセット・結合キー・ゲート条件)。§4 の逆引きと §1 のnull率事前計算はこのレジストリを入力に機械化する(パネル A-06)。
- KBが可能にすること: (a) 結合の網羅列挙(組み合わせの取りこぼしゼロ)、(b) null率の事前計算(ゲート判定が公開前に機械でできる — レジストリの「主要数値フィールド指定」を入力に)、(c) 統計更新時の**全ページ限界費用ゼロの再生成**(ただし文章ブロックの変更を含む再生成は検品キュー必須 — 33章 §5 の二分規約)。

## 2. 展開規模の設計(数万ページ能力の内訳)

| 軸の階数 | 結合式の例 | 理論ページ数 | 公開見込み(ゲート後) |
| --- | --- | --- | --- |
| 1軸 | 職種(12)・施設(10)・資格(10)・県(47) | 〜100 | ほぼ全数(ハブ) |
| 2軸 | 県×職種(C2 実在粒度 47×2〜3)・施設×属性(10×10)・資格×県(10×47) | 〜1,000 | 数百(V2 25章 S/A群 ≈ 600) |
| 3軸 | 県×職種×年度(推移)・施設×職種×属性 | 〜10,000 | **原則ページ化しない**(年度はページ内の推移グラフ・表で吸収。属性はセクション統合 — V2 S-03 エンティティゲート) |
| 時間軸展開 | 回次×試験(C4)・年度×改定(B4)・月次×求人倍率(C3) | 年数×対象 | 年次ハブに集約(「2027年版」の年版URL乱立をしない) |

- **時間軸と V2 25章 S案の棲み分け(パネル A-05)**: 25章 #40(回次×合格点)は「推移ハブ1ページ+**検索需要が実測できた直近2〜3回のみ**個別ページ」、#41(年度×日程)・#34(年度×カレンダー)は「**同一URLを年次上書き更新**(年版URLを増やさない)」と定義する。25章の想定数はこの規約で読み替える(本章が25章の公開粒度を制約する唯一の箇所であり、他は判断を変えない)。

- **設計判断: 「数万ページを作れる構造」と「数百〜数千ページを公開する運用」を分離する**。3軸以上の直積は原理的に可能だがシンコンテンツ量産になるため、**ページは2軸まで・3軸目以降はページ内のデータ表現(グラフ・表・セクション)で吸収**を規約とする。将来、GSC実測で特定の3軸クエリ(例「東京 特養 夜勤 給料」)の需要独立性が確認された場合のみ個別に開放する(V2 25章 §4 条件5 と同じ実測主義)。
- 成熟時の公開ページ総数見込みは V2 25章 §5(1,500〜1,900)を変えない。**KB化で増えるのは「ページ数」ではなく「1ページあたりのデータ密度と再生成能力」**。

## 3. 内部リンクの自動生成(E3 関係グラフ)

手書きの対応表(V1 06章 internal-link-map)を、E3 グラフからのルールベース導出に置換する:

```
導出ルール(例)
1. works-at:   職種ページ → その職種が働く施設ページ(weight順に3件)
2. requires:   施設・職種ページ → 必要資格ページ(資格CVへの動線)
3. leads-to:   資格ページ → 次に目指す資格(キャリアパス #71)
4. compares-with: 施設ページ → 比較ページ(#58。ペアはグラフで管理)
5. explains:   統計値・制度の登場箇所 → 用語集D1・加算B3 の解説(ページ内ボックス)
6. 記事⋈エンティティ: 記事frontmatterの entityRefs → 該当ハブ・pSEOページ(V2 26章 §3 ④の一般化)
```

- 利点: (a) リンク切れ・リンク漏れの機械検出(グラフとページの整合CI)、(b) 新ページ公開時に**既存ページ側の関連リンクも自動で更新**される(片方向リンクの根絶 — 有向関係の逆リンクはビルド導出。31章 §3 E の方向規約)、(c) 回遊設計の一貫性。
- **機械導出の副作用の統制(パネル S-04・A-07)**:
  1. **文脈リンク要件との区別**: V2 25章 §4 条件2 の「関連ガイド3本への文脈リンク」は**本文内アンカーに限定**し、E3 由来の自動関連ボックスはカウントしない(定型リンクブロックはボイラープレートでありゲート充足にならない)。
  2. **同型リンクの回避**: 決定的な weight 順だけで並べず、**ページ側のデータで並べ替える**(例: その県の統計値に基づく上位施設種別 — ページごとに異なる根拠でリンク先が変わる)。アンカーテキストは A6/D1 の同義語彙から2〜3パターンをローテーション(完全一致アンカーの大量反復を避ける — V1 03章 規則5)。
  3. **リンクグラフの安定と監視**: E3 導出結果はスナップショットとしてコミットし、**リンク差分が全リンクの5%を超えるビルドは人間レビューを要求**。同一ターゲットへの被リンク集中(in-degree 分布)をCIで監視する。
  4. **記事系の役割分担**: E3 は internal-linker への**候補プール供給のみ**。採否・位置・アンカーは internal-linker の文脈判断(「H2末尾で読者の次の疑問に答えるリンクだけ張る」等の既存判断基準)を維持する。pSEOページはエージェントを通らないため、上記1〜3の機械統制がそのまま適用される。
- 手動オーバーライドは残す(編集判断が必要な文脈リンク)。**自動6割・手動4割**を想定し、V1 03章の内部リンク規則(3クリック・アンカーテキスト)はそのまま適用される。

## 4. kbRef 逆引きインデックス — 「更新に強い」の実体

ビルド時に **KBレコード→使用URL の逆引き表**を2系統から生成する(パネル A-06): **(a) 記事**=生 Markdown の `<!--kb:...-->` コメント走査(Nuxt Content はレンダリング時にHTMLコメントを落とすため、**レンダリングASTではなく生ファイルを走査する独立ビルドスクリプト**として実装)、**(b) pSEOページ**=E7 レジストリ(結合式)からの導出。

```
salary-hokkaido-kaigo-2025 → /kyuryo/pref/hokkaido/, /kyuryo/pref/hokkaido/kaigoshoku/,
                              /tenshoku/guide/hokkaido-tenshoku/(記事内kbRef), sitemap-stats.xml
```

- **用途1: 制度改正・統計更新の影響特定**: B4 イベントの `affectedDatasets` → 逆引き → 影響URL一覧が機械で出る。「改定があったがどの記事を直すべきか分からない」を構造的に解消(1人運営の最重要保険)。
- **用途2: stale 伝播**: stale レコード(32章 §2)を参照中のページ一覧を月次レポート化。
- **用途3: 更新シグナルの誠実な運用**: `updatedAt`(sitemap lastmod・dateModified)を進めるのは**「本文・主要データブロックが kbRef で参照するレコードの値が変わった場合」のみ**と定義する(パネル S-03)。関連リンクボックスの自動更新・`checkedAt` のみの更新(値不変)・グラフの再描画(データ不変)ではページHTMLが変わっても日付を進めない。**中身の変わらない日付更新の自動化は鮮度偽装**であり、kb-lint の「updatedAt 誠実性」チェック(32章 §4)で機械担保する。

## 5. 構造化データ(JSON-LD)のKB駆動生成

| schema.org 型 | 源泉KB | 適用 |
| --- | --- | --- |
| Article / BreadcrumbList | 成果物メタ+URL階層 | 全ページ(V1 05章 §2 継承) |
| Person(監修者) | A5 authors(sameAs 含む) | 監修者ボックス・/supervisor/ |
| FAQPage | D2 faq(jsonLdEligible=true のみ。**同一FAQの複数ページ重複出力はCIで禁止**) | 記事・ハブの FAQブロック |
| **Dataset** | C1 stat-sources+C2/C3 統計 | **調査単位(C1 レコード単位)の出典ハブ/アーカイブページに1回だけ出力**し、`isBasedOn`・`sameAs`(e-Stat/公表元URL)・`creator`(政府機関)・`license` を正しく設定(政府統計を自サイトのデータセットと誤認させる帰属表示をしない — パネル S-02)。個別pSEOページは V1 通り Article+出典リンクを維持。**本行は V1 05章 §2「Dataset は使わない」の明示的上書き**(README 上書き表に記載) |
| Quiz / Question | D3 ドリル | V1 05章 §2 の方針継承(表示は期待しない) |
| SpecialAnnouncement 等の時事系 | 使わない | 過剰マークアップはスパムシグナル(最小主義 — V1 05章) |

- schema-generator(既存Agent)の生成元をKB参照に統一することで、**マークアップと本文の数値不一致が構造的に起きなくなる**(同じレコードから両方を生成するため)。
