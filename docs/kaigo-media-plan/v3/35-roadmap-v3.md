# 35. ロードマップ V3 — KB構築の段階導入

> **V2 23章の Priority(P1〜P10)・月次時間予算・ロードマップの時系列は変えない**。V3 は各 Priority の「実装様式」にKB観点を追加するだけである(KB化を理由に公開時期を遅らせない)。本章は V2 23章への追補として読む。

## 1. 大原則 — KBは「先に全部作る」ものではない

1. **YAGNIゲート**: 消費者(そのデータを使うページ・機能)が**2つ以上確定するまでデータセットを作らない**。「いつか使う」でKBを太らせない(1人運営の防衛線)。
2. **既存Priorityへの同乗**: KB構築は独立プロジェクトにせず、P6(統計)・P8(ドリル)など**どうせデータを作るタイミングでKB規約に沿って作る**。純増コストを規約適合分(+10〜15%)に抑える。
3. **ゲートは充実度に比例して締める**: F-1原則(KBにない事実は書けない)は最初から強制しない(KBが薄い段階では記事が書けなくなる)。warning → 対象拡大 → error 化の順(33章 §6)。

- **開発**の追加工数 合計 **約23〜36h**(2026年7月〜2027年末に分散)。V2 23章の月次時間予算「開発 5〜8h」の枠内で吸収可能(KB-0 のみ P2〜P3 期の開発枠を使う)。**Phase 3 は P8/P9 と KB-2 の開発が重なるため上限側に張り付く**(月6.3〜8.4h) — 超過月は KB-2 側を翌月へずらす(KB はすべて V2 Priority の従属)。
- **開発とは別に、KBデータ整備・人間承認の人的工数が存在する**(パネル K-01 の検算: KB-2 の用語・FAQ・加算・エンティティの承認だけで累計25〜40h規模)。これは §3 の月次枠で管理する。

## 3. 月次時間予算との整合(V2 23章 §2 への追補)

- **V2 23章 §2 の表に「KB整備・承認」行を追加する(月2〜3h)**(パネル K-01 — pSEO独自ブロックを V2 M-04 で別建てしたのと同型の扱い)。この枠に含まれるもの: kb-diff の承認レビュー・**月次KBレビュー(平常月60〜90分・繁忙月〔3〜4月〕2〜4h — 32章 §5-3)**・source-report の週次確認(10〜15分×4)。繁忙月は「リライト・更新」枠(4〜8h)から融通する。
- V2 23章 §2 の合計は 50〜80h → **52〜83h** に更新して読む(上限87hに対し残余4h〜。予算逼迫時の減速順は「ドリル→pSEO→記事」の前に**KB新規データセットの凍結**を最初に行う)。
- KB起案・検証はAI(kb-batch)が行うため、人間の関与は**承認 diff レビューのみ**。月次の承認が2〜3h枠を超え続ける場合はデータセットを縮小し、承認を表単位の一括方式(32章 §3)へ寄せる(YAGNIゲートの運用形)。
- **KB-2 マイルストーンの逆算修正**: 月2〜3h × 6ヶ月=12〜18h で承認できる規模に合わせ、2027年6月時点の目標を「**用語・FAQ 各100〜150件・加算マスタは処遇改善系(率型)の現行分のみ** verified」に縮小する(初稿の「用語200・FAQ200・加算現行分」は承認工数25〜40hで枠外 — パネル K-01/V3-04)。

## 5. リスクと撤退basis(V3 固有)

| リスク | 対応 |
| --- | --- |
| **R-15 KB過剰投資**: データ整備が目的化し、コンテンツ公開が遅れる | YAGNIゲート(§1)+「KB化を理由に公開を遅らせない」原則。KBフェーズはすべて V2 Priority の従属であり、単独でスケジュールを持たない |
| **R-16 KB単一障害点**: KBの誤りが全成果物に波及する(誤った加算率が記事・ツール・pSEOへ一斉展開) | 32章の人間承認ゲート+YMYL系の全件レビュー+kbRef 逆引きで**誤り発覚時の影響範囲特定と一括修正も高速**(波及は諸刃であることを認識して統治する) |
| **R-17 規約倒れ**: KbMeta・kbRef の記入が形骸化する | 機械CI(kb-lint)で強制。人間の善意に依存する規約を作らない(チェックできない規約は規約にしない) |

## 6. 5〜10年の視点(V3 の到達点)

- **3年後**: 統計5年分の推移・制度改正タイムライン・用語/FAQ 1,000件・ドリル3,000問+正答率 — 「介護の働き手データなら国内で最も構造化されているサイト」。引用・被リンクが自然に集まる状態(V2 R-09 の恒常化)。
- **5年後**: KBを源泉に、サイト内検索・パーソナルレコメンド・チャット(KB-4)を**必要になった時に**後付けできる。複数人体制・CMS移行(V1 06章 §5)にも KbMeta 構造のまま移行可能。
- **10年後**: サイトの形態(SSG/アプリ/音声/次世代検索)が何に変わっても、**KB=出典つき構造化知識は形態非依存の資産**として残る。これが「記事を書くAI」ではなく「知識基盤を成長させるプラットフォーム」の意味である。
