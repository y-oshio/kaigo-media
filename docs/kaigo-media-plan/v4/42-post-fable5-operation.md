# 42章 Fable5卒業後の運用設計 — 軽量モデル中心体制の作業分担と禁止事項

- 作成日: 2026-07-05
- 前提: 今後の Claude Code 作業は軽量モデル(Sonnet / Haiku 相当)が既定。高性能モデルは例外的に呼び戻す
- 原理(40章 §2): 安全性はモデルの賢さではなく**ゲートの網羅性**で担保する。
  ゲート未整備の領域を軽量モデルに渡さない

## §1 作業のモデル階層マトリクス

### 軽量モデルへ渡す作業(既定)

| 作業 | 条件(この条件を欠くなら渡さない) |
| --- | --- |
| 記事生成(ai-company パイプライン実行) | P5品質CI稼働済み+49章の記事設計あり+人間検収(§5) |
| 手順書のあるコード変更(Content v3移行=47章、routeRules、ゲート一元化、og:image) | 手順書+計画提示→承認+`npm run check` 通過 |
| lint/typecheck エラーの修正、依存の patch/minor 更新 | check 通過が完了条件 |
| KBレコードの**起案**(draft 投入) | status は draft 固定。昇格は人間のみ(不変) |
| 既存記事のリライト起案、内部リンク追加起案 | 43章 Rewrite Planner の判断基準に従う |
| GA4/GSC エクスポートの集計・レポート作成 | data/ops/ に保存された実測データのみを入力とする(§6) |
| 定型ページ追加(STATIC_PAGES 登録を含むチェックリスト遵守) | sitemap 規約コメントに従う |

### 高性能モデルへ戻す作業(例外)

- **スキーマ・ゲート・CIそのものの変更**(publishBlockers、kb-lint 規則、公開ゲート条件、CI yaml の検査内容)— ゲートを書き換える作業をゲートで守ることはできないため
- 新データセットの抽出パイプライン設計(P9の新職種・新調査の extract/build/verify 設計)
- KB-2/KB-3 のスキーマ設計、V3設計書の改訂
- P8ドリルの戦略判断、Phase 4 拡張設計、SERP環境変化(AI Overviews等)への戦略転換
- **インシデント対応**: 誤った統計値・事実の公開が発覚した場合の原因分析と再発防止設計
- 43章のSEO運営AIが出した「矛盾する提案」の裁定

### 人間(ユーザー)にしかできない作業(不変・モデル不問)

- KB status の verified/published への昇格(`npm run kb-promote`)
- 記事の公開判断(§5 検収)・/about/ 実在情報・監修者契約・A8/案件登録・過去問許諾
- ドメイン/DNS/Vercel/GSC の操作、支出を伴う全判断

## §2 Claude Code での作業分担の型

1. **セッション開始**: memory → 40章§5 の読み順で文脈復元。v4 の意思決定(41章)を再議論しない
2. **実装系タスク**: 計画提示 → ユーザー承認 → 実装 → `npm run check` → 結果を数値で報告(「たぶん動く」禁止)
3. **記事系タスク**: 46章の編集部パイプラインに従う。1セッション1記事を上限とする(検収負荷を人間側で平準化)
4. **分析系タスク**: 43章の該当Agent定義をプロンプトとして使う。入力データの実在(data/ops/ のファイル)を最初に確認し、無ければ「データ取得依頼」を出して止まる
5. **迷ったら止まる**: 41章・V3・本章に判断がない事項は、推測で進めず選択肢+推奨を提示して停止

## §3 ai-company の使い方(V2化までの経過措置)

- 現行 ai-company(9 Agent)は **46章の V2 組織へ改組するまで「凍結中」**。
  例外: 47〜49章の成果物が揃った後の「ドライラン1本」(44章 M2)のみ現行資産を使ってよい
- `templates/kakomon-explanation.md` は凍結領域(/kakomon/)前提のため**使用禁止**(V2で凍結済み)
- パイプライン実行時のモデル割当は 46章 §4 の表が正

## §4 記事生成時の安全ゲート(直列・全通過が公開の必要条件)

1. **G1 設計ゲート**: 記事設計(構成+KB参照マップ)が存在するか。無ければ書き始めない
2. **G2 F-1ゲート**: 本文中の検証可能な事実は全て KB レコード(verified/published)への参照を持つか。
   KBにない事実は書けない。体験談は exp マーカー必須
3. **G3 出典ゲート**: frontmatter sources 全件に name+checkedAt(publishBlockers と同一規則)
4. **G4 機械ゲート**: `npm run check`(content-lint: banned phrases・/go/→prRelated 検査を含む)
5. **G5 人間検収**: §5 のチェックリスト。**公開(コミット)操作は人間の承認後のみ**
6. **G6 リーガル定数**: /go/ リンクは rel="sponsored nofollow" 機械付与の経路(ProseA/CtaSlot)のみ使用

## §5 人間が必ず確認する箇所(検収チェックリスト最小版)

- [ ] 統計値・金額・制度名・年度の全件を出典と突合した(サンプリング不可・全件)
- [ ] 断定的な医療・法律・金銭アドバイスになっている文がない(YMYL)
- [ ] 実在しない体験・人物・事業所への言及がない
- [ ] タイトル・description が本文内容を誇張していない
- [ ] 内部リンク先が実在し、意図したページである
- [ ] CTA の表示条件(案件登録済みか)が正しい

## §6 やってはいけない運用(禁止事項)

1. **軽量モデルにゲート自体を変更させない**(publishBlockers・lint規則・公開ゲート値・CI定義)
2. **CIが赤のまま公開・コミットしない**(「あとで直す」を許すとゲートが飾りになる)
3. **KBを経由しない事実記述**(F-1違反)。「一般に知られている」は根拠にならない
4. **計測値の捏造・推定**: GSC/GA4 の数値は data/ops/ の実測エクスポートのみ。
   「おそらく順位は〜」を数値として文書に書かない(F-1 の計測データ拡張)
5. **サンプル記事・仮記事・架空案件のコミット**(既存規約の継続)
6. **status 昇格の自動化・代行**(モデルがどれだけ確信していても不可)
7. **一括大量公開**: 記事は検収済みのものから逐次公開。未検収記事のバッチ公開禁止
8. **`pkill -f 'nuxt dev'` の実行禁止**(セッション自殺)。devサーバ停止は `fuser -k 3000/tcp`
9. **git 操作は kaigo-media 内限定**・force push 禁止・main 直 push は CI 通過後のみ
10. **凍結領域(/kakomon/)の解凍**は許諾書面の確認をユーザーが宣言した場合のみ
