# 26. オリジナル学習コンテンツ設計(著作権リスクゼロの問題集)

> V1 03章の `/kakomon/` URL体系・04章 F1・06章 §2-1 の運用を上書きする。データスキーマ自体は V1 の `KakomonQuestion`(origin:'original')を流用し、**情報設計のやり直しは発生しない**(V1 レビュー C-2 でプランBとして設計済みだったものを主力に昇格・拡張する)。

## 1. 設計原則

1. **クリーンルーム生成(R-10 対応・本設計の生命線)**: 問題の生成工程で実在の過去問本文・選択肢・市販問題集のテキストを**一切入力に使わない**。使ってよい素材は (a) 試験センター公表の出題基準の**論点名**(事実)、(b) 法令・厚労省資料・公的テキストの知識内容、(c) 自作の論点マスタのみ。
2. **事実と著作物の峻別(R-11 対応)**: 合格点・合格率・日程・科目名・出題基準の項目名は事実として利用可。出題基準の文章の逐語コピー・正答表の複製はしない。
3. **全問監修必須(R-13 対応)**: オリジナル問題は「出典に正解を委ねられない」= 正誤の責任が全て自サイトにある。監修者チェックなしの問題は1問も公開しない(過去問解説より監修必須度が高い)。
4. **1問1ページにしない**: オリジナル問題には「第N回 問N」の指名検索が存在しないため、1問1ページはシンコンテンツ化するだけ。**論点ページ(問題5〜10問+論点解説)を最小公開単位**とする。

## 2. コンテンツフォーマット(7形式)

| 形式 | 内容 | 対象クエリ(仮説) | Phase |
| --- | --- | --- | --- |
| F-A 一問一答(5択) | 本試験と同形式。論点ページに5〜10問 | 「(科目名) 問題」「介護福祉士 一問一答」 | P8(最優先) |
| F-B ○×スワイプ | スキマ時間向け。科目×100本 | 「介護福祉士 ○×」「アプリ 無料」 | P8+1 |
| F-C 確認テスト | 論点横断10問セット・採点つき | 「介護福祉士 確認テスト」 | P8+1 |
| F-D ケーススタディ | 事例文(オリジナル)+設問2〜3。実務感のある独自素材 | 「事例問題 練習」 | Phase 4 |
| F-E 模擬試験 | 125問フル・パート別採点(A/B/C 新制度対応) | 「介護福祉士 模試 無料」 | Phase 4 |
| F-F 制度クイズ | 介護保険・処遇改善など働き手向け制度理解(試験非依存=通年需要) | 「介護保険 クイズ」 | Phase 4 |
| F-G 実務シミュレーション | 分岐型(「このときどう対応する?」)。SNS拡散・被リンク素材 | 指名・SNS流入 | Phase 4(実験枠) |

- **初任者研修・実務者研修の修了試験対策(非公開試験)は競合がほぼ不在**(25章 #50)。介護福祉士と並ぶ優先ターゲットにする(講座資料請求CVにも最短)。

/shiken/kaigofukushishi/drill/<subject>/<topic>/  └ 論点ページ(最小公開単位: 問題5〜10問+論点解説)← 階層3で3クリック規則に適合

- V1 の内部リンク規則「1問ページのみ4クリック許容」は不要になる(論点ページは3クリック)。**V2 では全ページ3クリック以内に統一**。
- 論点ページの内部リンク: ①前後の論点 ②科目ページ ③関連論点(タグ) ④文脈に合う記事(科目→記事マップは V1 06章 internal-link-map を継承)。

## 4. 論点マスタとページ規格(V1 04章 F1 の差し替え = F1')

### 論点マスタ(タクソノミー)

- 試験センター公表の出題基準(大項目・中項目)を**参考に、自分の言葉で再構成した論点マップ**を作る(項目名の丸写しリストにしない — R-11)。
- 規模: 介護福祉士13科目 × 論点15〜30 = **約300論点**(=約300ページ、問題総数 1,800〜3,000問)。
- データ: `data/drill/topics.ts`(DrillTopic: examSlug / subjectSlug / topicSlug / name / summary / order / examBasisNote)+ `data/drill/<subject>/<topic>.json`(KakomonQuestion[] origin:'original')。

### 論点ページ規格(F1')

```
- H1: 「(論点名)の練習問題(全N問)|(科目名)」
- 論点解説(300字以上): この論点で問われる知識の要点・制度根拠 ← ページの独自価値の核
- 問題1〜N(タップで解答表示。正解・解説は初期HTMLに含めCSS制御 — V1 05章 §4 を継承)
  - 各問: 問題文 / 選択肢5 / 正解 / なぜ正解か(制度根拠つき) / 各誤答がなぜ違うか / 覚え方
- 正答率表示(自サイト集計。データが溜まるまで非表示)← 誰にもコピーできない独自データ
- 監修者ボックス(氏名・資格・監修日)
- 次の論点へ / 科目に戻る / 関連記事 / 弱CTA(講座資料請求 or LINE)
```

- **定量下限(品質CI: V1 ツール#6 に追加)**: 論点解説300字以上 / 1問あたり解説合計200字以上 / 誤答理由は全選択肢分 / 制度・法令名の明記1つ以上 / 監修者ID必須。下限を割るページは公開しない。

## 5. 制作パイプライン(drill-batch — ai-company への追加。24章参照)

| 工程 | 担当 | 内容 |
| --- | --- | --- |
| 1. 論点選定 | 人間(月次) | 論点マスタから当月分を選定(需要実測とドリル進捗で優先度) |
| 2. 作問(クリーンルーム) | AI | 論点の知識内容(法令・公的資料)だけを入力に、問題+選択肢+解説を生成。**過去問・市販問題集をプロンプトに入れることを機械的に禁止**(パイプラインが受け付けない) |
| 3. ファクトチェック | AI+人間 | 正解の制度根拠を一次資料で確認(現行制度か・改定予定はないか) |
| 4. 類似性チェック | 人間+監修者 | 市販問題集・公表されている試験関連資料と**人間が目視突合**し、偶然の酷似を検出したら問題を差し替え(R-10)。チェック済みフラグをデータに記録 |
| 5. 監修 | 監修者(科目適性で振り分け) | 全問。正誤・選択肢の紛らわしさ・制度改定の見落とし |
| 6. 公開 | 人間 | 論点単位で逐次公開。公開ゲート(インデックス率90%)は V1 04章 §3 を継承 |

- 目安工数: 1論点(5〜10問)= AI生成15分+人間検証・監修30〜60分【H-33】。月20〜30論点ペース → 13科目・300論点を約12ヶ月で完成(23章 Phase 3〜4)。

## 6. 「Googleに評価される」ための設計(SEO・E-E-A-T)

1. **検索意図の正直さ**: ドリルページのタイトル・メタに「過去問」を使わない(「過去問」で集客して別物を出すのは品質評価上も倫理上も悪手)。狙うのは「問題」「一問一答」「練習問題」「予想問題」「対策」。「過去問」需要には試験ハブの「合格点・出題傾向」ページで応える(そこからドリルへ内部誘導)。
2. **独自データの蓄積**: 正答率・よく間違う選択肢の集計は、実過去問サイトが持たない一次データ。「受験者の◯%がこの論点でつまずく」は引用・言及されうる素材(R-09 の被リンク施策と連動)。
3. **鮮度シグナル**: 制度改定(介護報酬・処遇改善は毎年度)のたびに影響論点を更新カレンダー(V1 06章 §4)で洗い出し、`updatedAt`+改定注記を更新。旧制度のまま放置された競合問題集との差別化点。
4. **構造化データ**: Quiz/Question マークアップは V1 05章 §2 の方針を継承(表示は期待しない・低コスト範囲のみ)。
5. **回遊とリピート**: 弱点ダッシュボード(localStorage・ログイン不要)+「次の論点」動線+LINE「毎日3問」配信(H-35)で、指名検索リピーターの不在を製品体験で補う。

## 7. CV接続

- 学習中ユーザーへの転職CTAはノイズ(V1 07章の判断を継承)。一次CTA = 講座資料請求(実務者研修比較=受験資格に直結)/ 二次 = LINE(毎日3問+合格後キャリア情報)。
- 修了試験対策(初任者・実務者)だけは講座選び直前の層なので資料請求CTAを標準配置。
- 合格発表(3月)後のLINE配信で転職導線へ — V1 の時間差モデルを縮小継続(H-35)。
