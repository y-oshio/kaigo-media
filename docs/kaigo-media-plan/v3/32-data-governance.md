# 32. データガバナンス — KBの規律・ライフサイクル・更新戦略

> V1 06章 §4(運用データ)と V2 R-10〜R-14 の運用を、KB全体の統治規約に一般化する。**KBの信頼性はサイトの信頼性の上限**である(KBが誤ればすべての成果物が誤る)。

## 1. 五大原則

1. **出典必須・捏造禁止**: L2ファクトの全レコードは `sourceIds` ≥1+`checkedAt` 必須。出典に存在しない粒度を推計で埋めない(null許容 — V1 06章の原則をKB全体へ)。**出典のないファクトはCIが弾く**。
2. **単一定義(Single Source of Truth)**: 同じ事実を2つのデータセットに書かない。参照はIDで行う(例: 施設の給与は A3 に書かず C2 を参照)。記事 frontmatter の出典も D4 参照へ移行し二重管理を解消(35章 KB-2)。
3. **履歴は消さない**: 統計は年度キーで追加、制度は `validFrom/validUntil` で並存、エンティティはgit履歴。「上書きで過去が消える」構造を作らない(推移・タイムライン成果物の源泉であり、5〜10年の複利の実体)。
4. **事実と見解の分離**: KBに入るのは検証可能な事実のみ。編集部の意見・体験談・評価(「働きやすい」等)は成果物(記事)側に置き、KBには入れない。境界例は `workStyle`(A3)のような「編集部整理」フィールドで、**出典不要な代わりに status: verified に人間承認が必須**。
5. **AIは起案、人間が承認**: KBへの書き込みは誰(AI/機械)が起案してもよいが、**status を verified/published に上げられるのは人間のみ**(YMYL系データセット B/C/D3 は必須。D1/D2 はサンプリング承認可 — §3)。

## 2. レコードライフサイクル

```
draft ──(出典突合OK)──→ verified ──(サイトが参照開始)──→ published
  ↑                                                        │
  └──(修正)── stale ←──(鮮度切れ検知: CI が自動遷移)─────────┘
                │
                └──(役目を終えた/誤りだった)──→ archived(削除はしない・IDは欠番)
```

- **stale の機械検知(品質CIの一部)**: (a) `checkedAt` + データセット既定の確認周期(§5 の表)を超過、(b) `validUntil` 超過、(c) 参照先の出典URLが404、のいずれかで自動的に stale へ。stale レコードを参照するページはビルド時に警告(重大データセット B3/C5 はビルド失敗)。
- **archived**: レコードは物理削除しない(IDの欠番運用 — V2 26章の問題ID規約と同じ)。誤りだった場合は `note` に理由を残す(「なぜ消えたか」も知識)。

## 3. 更新責任マトリクス(RACI)

| 工程 | 担当 | 内容 |
| --- | --- | --- |
| 監視・収集 | **source-manager(AI)+機械** | 公表カレンダー・官報・省庁サイトの監視、一次資料の取得、利用規約の確認(R-14チェックリスト) |
| 構造化・起案 | **knowledge-builder(AI)** | 一次資料→KBレコード案(diff形式)。既存レコードとの矛盾を明示 |
| 機械検証 | **品質CI(スクリプト)** | スキーマ・参照整合・鮮度・定義差混在(§4)。人間の目に入る前に機械で落とす |
| 意味検証 | **data-validator(AI)** | 数値の桁・単位・前年との乖離妥当性・出典の読み違い検査(機械検証で表現できない部分) |
| 承認 | **人間(運営者)** | diff をレビューして verified へ。YMYL系(制度・統計・試験・料率)は全件、D1/D2 は10件に1件のサンプリング+全件の出典リンク確認 |
| 監修 | **監修者** | D3(ドリル問題)のみ全問(V2 26章 §5 — KB化しても監修ゲートは不変) |

- 1人運営のため「承認」はすべて同一人物だが、**工程として分離しておくことが将来の複数人化(V1 06章 §5 の移行トリガー)への備え**になる。

## 4. 品質CI(V2 P5 の品質CIにKB検証を追加)

| チェック | 内容 | 失敗時 |
| --- | --- | --- |
| スキーマ検証 | 全データセット×Zod。KbMeta 必須フィールド・型 | コミット不可 |
| 参照整合 | sourceIds / lawRefs / entityRefs / slug の実在。孤児参照ゼロ | コミット不可 |
| 出典義務 | L2ファクトの sourceIds 空を検出 | コミット不可 |
| 鮮度検知 | §2 の stale 条件。checkedAt+周期超過の一覧を月次レポート | 警告(B3/C5 参照ページはビルド失敗) |
| 定義差混在 | 同一表・同一グラフに sourceId の異なる統計値を並べていないか(V1 06章 §3「2つの調査を同一表に並べない」の機械化) | ビルド失敗 |
| 回次・年度導出 | 試験回次・年度の手書きを検出し ExamRound / validFrom からの導出を強制(V2 K-03) | ビルド失敗 |
| ID不変 | 既存IDの変更・再利用を git diff で検出 | コミット不可 |
| F-1 突合(段階導入) | 記事・pSEO本文中の数値・制度記述が kbRef を持つか(33章 §4。まず統計値から) | 警告→Phase 3 以降エラー化 |

### 5-2. イベント駆動(source-manager の監視対象 — 33章)

- **官報・厚労省の告示/通知**(処遇改善の随時改定・補助金) → B3/B4 を更新 → 影響URLを逆引きして更新(34章 §4)。
- **法改正の成立・施行**(介護保険法等) → B1/B2/B4 → `validFrom` 付きの新レコード追加(施行日前に準備し、施行日にビルド)。
- **試験制度の変更発表**(パート合格制度のような大変更) → D3/C4/B4 → 該当コンテンツの改稿計画を knowledge-manager が起案。
- 監視の実装: 週1回の定例チェック(source-manager の巡回リスト=C1/D4 の公表元URL)。リアルタイム監視インフラは作らない(1人運営)。

### 5-3. 鮮度CI駆動(セーフティネット)

- カレンダー・イベントで拾えなかった陳腐化は §4 の stale 検知が拾う。**月次KBレビュー(30〜60分。V2 23章 §2「運用」枠に内包)**で stale 一覧を消化する。この時間で収まらない場合はデータセットの縮小を検討する(YAGNIゲート — 35章)。

## 6. 著作権・利用規約(V2 R-10〜R-14 のKB適用)

- **政府統計**: 公共データ利用規約(政府標準利用規約準拠)に基づき出典明記で利用可。e-Stat API を使う場合はAPI利用規約の確認を KB-1 時に実施(C1 に記録)。
- **法令**: 条文自体に著作権はないが、KBは要旨+参照で持つ(§31-4。保守性の問題であって権利の問題ではない)。
- **第三者公表データ(E5)**: R-14 チェックリスト(規約確認・checkedAt・年2回更新・「最新は公式で」注記)を通過したもののみ。
- **画像・図表(D5)**: ライセンスフィールド必須。出典不明画像は登録不可。図表は自作(元データ+生成仕様)を原則とする。
- **過去問**: KBに本文を入れない(31章 §4)。類似性チェック用の照合DB(V2 26章 §5 工程4・30条の4)は**KBとは別置き**(公開ビルドに含まれない `work/` 相当の内部データ)とし、混入をディレクトリ境界で防ぐ。
