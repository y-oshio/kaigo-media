# 30. Knowledge Base 戦略 — 「記事を書くサイト」から「知識基盤を成長させるプラットフォーム」へ

> V1 00章「戦略の核心」に思想レイヤーを追加する(V2 21章の4層エンジンは不変。本章はその**作り方**を変える)。

## 1. Knowledge Base の定義(本サイトにおける)

**Knowledge Base(KB)とは、「介護業界で働く人」に関する事実を、出典・確認日・安定ID・有効期間つきの構造化レコードとして git 管理した資産の総体**である。

```
KB(資産)                          成果物(射影)
┌─────────────────────┐    ┌─────────────────────────┐
│ L1 エンティティ       │    │ 記事(素材・数値の注入)      │
│  資格・職種・施設・県… │ →  │ pSEOページ(テンプレ×結合)   │
│ L2 ファクト           │    │ ドリル(論点マスタ×クリーンルーム)│
│  統計・制度・試験・法令│ →  │ ツール(料率・加算率の参照)    │
│  用語・FAQ           │    │ 比較表・ランキング・図表      │
│ L3 関係・独自データ    │ →  │ JSON-LD・内部リンク・OGP    │
│  関係グラフ・正答率・  │    │ LINE配信・独自調査レポート    │
│  診断集計・キーワード  │    │ (将来)RAG・チャット・検索    │
└─────────────────────┘    └─────────────────────────┘
```

- **L1 エンティティ層**: 業界の「名詞」のマスタ(資格・職種・施設形態・都道府県・監修者など)。変化が遅く、全成果物のジョイントキーになる。
- **L2 ファクト層**: 数値・規定・日付など「検証可能な事実」(統計値・加算率・合格点・法令要旨・用語定義・FAQ)。**全レコードに出典と確認日が必須**。時間軸(調査年・年度・有効期間)を持つ。
- **L3 関係・独自データ層**: エンティティ間の関係グラフ、自社でしか採れない一次データ(ドリル正答率・診断集計・独自調査)、運用台帳(キーワード・更新カレンダー)。**誰にもコピーできない差別化の実体**(V2 21章 §4 の一次データ戦略の置き場)。

### KBであるもの/ないもの

| KB である | KB でない |
| --- | --- |
| `data/` 配下の構造化レコード(31章の全データセット) | 記事本文の文章(=成果物) |
| 記事 frontmatter の構造化部分(出典・監修・クエリ) | 体験談・意見・編集部の見解(=成果物。事実と見解の分離 — 32章) |
| 図表の「元データ+生成仕様」 | 書き出されたSVG/画像ファイルそのもの(=ビルド成果物) |
| ドリルの論点マスタ・問題・正答率 | ページのHTML |

### 物理形態(V1 06章の方針を継承)

**KB = git 管理の TS/JSON ファイル群。実DB・外部SaaSは導入しない。** V1 06章の `data/` はKBの原型として正しく設計されており(出典マスタ・checkedAt・null許容・複合キー履歴)、V3 はこれを**全データセットに一般化して規約化**する(31章・32章)。規模拡大時の SQLite 移行パス(06章 §5)も不変 — KBのスキーマはそのまま DDL 化できる形を保つ。

## 2. 思想の転換 — 3つの原則

### F-1. 「KBにない事実は公開物に書けない」(最重要原則)

数値・統計・制度規定・法令・試験情報・費用・日付など**検証可能な事実**は、先にKBへ登録(出典つき)してからでないと記事・ページに書けない。逆に言えば、**記事制作中に調べた事実はすべてKB登録という形で資産化される**(調査の使い捨てを禁止する)。

- 対象: 数値全般・制度/法令の規定・試験事実・費用相場・固有名詞の定義。
- 対象外: 体験談・取材コメント・編集部の見解・文章表現(これらは成果物側の価値)。
- 効果: 同じ事実の再調査ゼロ/制度改定時に「どの記事に影響するか」を機械的に逆引き(34章 §4)/ファクトチェックが「Web照合」から「KB照合」になり高速化・機械化(33章)。

### F-2. 「成果物は再生成可能でなければならない」

KBが更新されたら、それを参照する成果物は(人間の書き直しなしに)再生成・再ビルドできること。統計の年次更新で数百ページが一括で新しくなるのは pSEO だけでなく、記事内の統計値・図表・比較表も同様にする(kbRef 参照 — 33章 §4)。**記事の文章は人間+AIの資産、事実はKBの資産**、と分離することで両方の寿命を最大化する。

### F-3. 「KBは検索エンジンにもAIにも同じものを見せる」

KBの1レコードは、単体で読んで意味が通る(自己完結)・出典がある・IDで参照できる。この規約(31章 §6)を守る限り、同じデータが (a) SSGビルドの入力、(b) JSON-LD(FAQPage/Dataset等)の源泉、(c) 将来のRAGのチャンク、の3役を無改修で務める。**RAGは実装しないが、RAG対応可能な形だけ先に決めておく**(31章 §6)。

## 3. なぜデータを資産化するのか(判断基準との対応)

| 判断基準(V2憲法) | KB中心設計の寄与 |
| --- | --- |
| Googleアップデート耐性 | コンテンツの独自性の源泉が「文章の上手さ」から「独自データ+鮮度」に移る。ヘルプフルコンテンツ観点で最も燃えにくい資産(一次データ・構造化・最新)に投資が集中する |
| 著作権リスク低 | KBは「事実+出典」しか持たない設計(事実に著作権なし・表現は自作)。R-10〜R-14 の防波堤がデータ層で一元化される |
| AIで継続運営 | AIの弱点(事実の幻覚・鮮度)をKBが補い、AIの強み(文章生成・構造変換)だけを使う分業になる。プロンプトに注入する事実が常にKB由来=検証済み |
| 1人で現実的 | 調査の再利用で記事あたり工数が逓減する。制度改定対応が「影響レコード更新→再ビルド」に縮む。**ただしKB維持自体の工数は新たに発生する**(35章で時間予算に組み込み。過剰装備の禁止=YAGNIゲート) |

- **5〜10年の視点**: 記事は3〜5年で書き直しになるが、エンティティマスタ・統計履歴・制度改正タイムライン・用語定義は蓄積するほど価値が増す(推移グラフ・「10年の変遷」コンテンツ・引用される統計アーカイブ)。サイトを閉じても KB は次の事業に持ち出せる。

| 成果物 | 源泉データセット(31章のID) | 生成方式 | 状態 |
| --- | --- | --- | --- |
| 記事(C2〜C6) | 全L1/L2(素材注入)+E1 キーワード | ai-company article-batch(KB対応版 — 33章) | 稼働済→KB対応はP6〜 |
| pSEO 統計ページ(25章 A〜G群) | C2 給与統計・C3 労働統計・A1〜A4 エンティティ | テンプレ×KB結合(34章) | P6/P9 |
| 学習ドリル(V2 26章) | D3 試験・論点・問題 | drill-batch(クリーンルーム — V2 26章 §5) | P8 |
| ツール(計算機・判定) | C5 税/料率・B3 加算・A1 資格要件 | SSG+クライアント計算(V2 25章 T9) | P6〜 |
| 比較表(施設ペア45通り・資格ペア・職種ペア) | A1〜A3+C2(給与差)+E3 関係 | 比較テンプレ×エンティティペア | Phase 3 |
| ランキング(給料の高い県・資格難易度順 等) | C2〜C4 統計 | ソート射影(#17 等) | P9 |
| FAQブロック+FAQPage JSON-LD | D2 FAQ | 記事・ハブページへの埋め込み+構造化データ | Phase 2〜 |
| 図表(SVGグラフ・推移・地図) | C2〜C4 統計+D5 生成仕様 | ビルド時SVG生成(統計更新で自動再生成) | P6〜 |
| タイムライン(制度改正の変遷・試験制度の変遷) | B4 制度改正イベント | 年表テンプレ | Phase 3 |
| チェックリスト(入職前確認・転職準備 等) | A3 施設+D2 FAQ+B2 制度 | リストテンプレ(#60・#105) | Phase 3 |
| 診断(8タイプ・受験資格判定・準備度) | E7 診断定義+A1〜A3+E3 関係 | 質問分岐×エンティティマッピング | P7〜 |
| 内部リンク・関連コンテンツ | E3 エンティティ関係グラフ | ルールベース導出(34章 §3。手書き対応表を置換) | Phase 2〜 |
| JSON-LD(Article/FAQPage/Dataset/Person/Breadcrumb/Quiz) | 各成果物+A5 監修者+C1 出典 | schema-generator(KB参照に変更 — 33章) | 稼働済→拡張 |
| OGP画像(タイトル+データハイライト) | 成果物メタ+C2 統計 | ビルド時生成(V2 P10) | Phase 4 |
| LINE配信(毎日3問・制度改定速報) | D3 問題+B4 イベント | 自動ローテーション(V2 26章 §6-5) | P8〜 |
| 独自調査レポート(年次「介護労働白書」風 #106) | L3 独自データ+C2〜C3 | 集計+編集(被リンク施策) | Phase 3〜 |
| (将来)RAGチャット・サイト内検索・レコメンド・質問応答 | KB全体のJSONLエクスポート | **実装しない。構造のみ対応**(31章 §6) | 将来(35章 KB-4) |

- 1つのKB更新が多数の成果物に波及する(例: 賃金統計の年次更新 → 統計ページ群・記事内数値・図表・ランキング・比較表・OGP が一括再生成)。**これが「データを資産化する」の実利**であり、34章で機構を定義する。
