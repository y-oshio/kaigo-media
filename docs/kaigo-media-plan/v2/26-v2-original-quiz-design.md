# 26. オリジナル学習コンテンツ設計(著作権リスクを最小化した問題集)

> V1 03章の `/kakomon/` URL体系・04章 F1・06章 §2-1 の運用を上書きする。データスキーマ自体は V1 の `KakomonQuestion`(origin:'original')を流用し、**情報設計のやり直しは発生しない**(V1 レビュー C-2 でプランBとして設計済みだったものを主力に昇格・拡張する)。
> 表題は初稿の「リスクゼロ」から「最小化」に改めた: クリーンルーム生成でも残存リスク(R-10)はゼロにならないため(§1-1 注記)。

## 1. 設計原則

1. **クリーンルーム生成(R-10 対応・本設計の生命線)**: 問題の生成工程で実在の過去問本文・選択肢・市販問題集のテキストを**一切入力に使わない**。使ってよい素材は (a) 試験センター公表の出題基準の**論点名**(事実)、(b) 法令・厚労省資料・公的テキストの知識内容、(c) 自作の論点マスタのみ。
   - **注記(残存リスク)**: クリーンルームは「プロンプトに入れない」ことしか担保しない。生成AIの事前学習データに過去問・市販問題集が含まれる場合、**非入力でも依拠性が認められうる**(文化庁「AIと著作権に関する考え方」2024年3月)。したがってクリーンルームは必要条件であって十分条件ではなく、§5 工程4 の類似性チェックとセットで初めて「最小化」になる。
2. **事実と著作物の峻別(R-11 対応)**: 合格点・合格率・日程・科目名・出題基準の項目名は事実として利用可。出題基準の文章の逐語コピー・試験直後の正答表の複製はしない(公式正答公表後の「問題文を伴わない正答番号一覧」の扱いは 22章 R-11(b) に従う)。
3. **全問監修必須(R-13 対応)**: オリジナル問題は「出典に正解を委ねられない」= 正誤の責任が全て自サイトにある。監修者チェックなしの問題は1問も公開しない(過去問解説より監修必須度が高い)。
4. **1問1ページにしない**: オリジナル問題には「第N回 問N」の指名検索が存在しないため、1問1ページはシンコンテンツ化するだけ。**論点ページ(問題5〜10問+論点解説)を最小公開単位**とする。

## 2. コンテンツフォーマット(7形式)

| 形式 | 内容 | 対象クエリ(仮説) | Phase |
| --- | --- | --- | --- |
| F-A 一問一答(5択) | 本試験と同形式。論点ページに5〜10問 | 「(科目名) 問題」「介護福祉士 一問一答」 | P8(最優先) |
| F-B ○×スワイプ | スキマ時間向け。科目×100本 | 「介護福祉士 ○×」「アプリ 無料」 | P8+1 |
| F-C 確認テスト | 論点横断10問セット・採点つき | 「介護福祉士 確認テスト」 | P8+1 |
| F-D ケーススタディ | 事例文(オリジナル)+設問2〜3。実務感のある独自素材 | 「事例問題 練習」 | Phase 4 |
| F-E 模擬試験 | 125問フル・パート別採点(A/B/C 新制度対応) | 「介護福祉士 模試 無料」 | Phase 4 |
| F-F 制度クイズ | 介護保険・処遇改善など働き手向け制度理解(試験非依存=通年需要) | 「介護保険 クイズ」 | Phase 4 |
| F-G 実務シミュレーション | 分岐型(「このときどう対応する?」)。SNS拡散・被リンク素材 | 指名・SNS流入 | Phase 4(実験枠) |

- **修了試験対策(25章 #50)は初任者研修を中心に据える**。初任者研修の修了評価は必須だが各スクール作成・非公開(合格率ほぼ100%の確認テスト)であり、**実務者研修は修了評価の実施自体がスクール裁量で「試験がない場合がある」**——この事実をコンテンツ内に明記し、「対策」需要の実在は T-0c 実測を公開の前提条件(25章 §4 条件4)とする。講座資料請求CVに最短である位置づけは維持。
- **ケアマネへの横展開時の注意**: ケアマネ本試験は**五肢複択(2つ・3つ選ぶ)形式**のため、F-A(五択一問一答)の単純流用はできない。複数選択用のテンプレート派生(出題形式・採点ロジック)が必要(Phase 4 の設計項目)。

## 3. URL・情報設計(V1 03章 C1 の差し替え)

```
/shiken/                                  試験・学習ハブ(C1' の入口)
/shiken/kaigofukushishi/                  └ 介護福祉士ハブ(日程・合格点推移・パート合格制度・科目一覧)※事実データ
/shiken/kaigofukushishi/goukakuten/       └ 合格点・合格率の推移(ExamRound データ駆動)
/shiken/kaigofukushishi/drill/            └ ドリル目次(13科目)
/shiken/kaigofukushishi/drill/<subject>/  └ 科目ページ(論点一覧+科目解説+進捗)
/shiken/kaigofukushishi/drill/<subject>/<topic>/  └ 論点ページ(最小公開単位: 問題5〜10問+論点解説)
/shiken/kaigofukushishi/moshi/            └ 模試(Phase 4)
/shiken/caremane/...                      └ ケアマネ(§2 の複択形式派生が前提)
/shiken/shoninsha/...                     └ 初任者研修(修了試験対策)
/kakomon/                                 【凍結・予約】正式許諾取得時のみ有効化(V1 03章の設計をそのまま使う)
```

- **クリック深度とURL階層を区別して規定する**: 論点ページのURL階層は5だが、資格ハブが13科目へ直接リンクするため**クリック深度はトップ→資格ハブ→科目→論点の3**(ドリル目次はハブ内セクション扱い)。URL階層5は V1 03章「階層は原則3」に対する**明示的な例外**とし、パンくず(BreadcrumbList)と正規URLで担保する。V1 の「1問ページのみ4クリック許容」規則は失効し、**V2 では全ページ・クリック深度3以内に統一**。
- 論点ページの内部リンク: ①前後の論点 ②科目ページ ③関連論点(`DrillTopic.tags`) ④文脈に合う記事(科目→記事マップは V1 06章 internal-link-map を継承)。
- **データ配置は `data/drill/` に統一**し、V1 06章 §1 の `data/kakomon/drills/` を上書きする(理由: ドリルは V2 の主力エンジンであり、凍結中の kakomon 配下に置くと「オプション扱い」と混同される。実装済みの `data/kakomon/` はオプション用にそのまま残す)。

## 4. 論点マスタとページ規格(V1 04章 F1 の差し替え = F1')

### 論点マスタ(タクソノミー)

- 試験センター公表の出題基準(大項目・中項目)を**参考に、自分の言葉で再構成した論点マップ**を作る(項目名の丸写しリストにしない — R-11)。
- 規模: 介護福祉士13科目 × 論点15〜30 = **約300論点**(=約300ページ、問題総数 1,800〜3,000問)。完成ペースは23章の月次時間予算から逆算し**月15論点前後・16〜20ヶ月**(初稿の「12ヶ月」は監修工数の検算で不成立となり修正)。

### データ仕様(型定義 — 実装時は `types/` に置く)

```ts
/** 論点マスタ: data/drill/topics.ts */
interface DrillTopic {
  examSlug: string          // 'kaigofukushishi' 等
  subjectSlug: string
  topicSlug: string
  name: string              // 論点名(自分の言葉。出題基準の逐語不可)
  summary: string           // 論点解説の要旨(科目ページの一覧に表示)
  order: number
  tags: string[]            // 関連論点リンク(§3 内部リンク③)の素
  examBasisNote?: string    // 出題基準のどの項目に対応するかのメモ(内部用・非表示)
  status: 'draft' | 'review' | 'published'
  updatedAt: string         // 制度改定時の鮮度シグナル(§6-3)
  publishedAt?: string
}

/** 問題(KakomonQuestion への V2 追加フィールド) */
//  id: string を必須化する。例 'kfs-<subject>-<topic>-001'
//  - 差し替え時は欠番にして ID を再利用しない(localStorage 履歴・正答率集計が壊れるため)
//  - similarityCheckedAt?: string を追加(§5 工程4 のチェック済み記録)
```

- 問題データ: `data/drill/<subject>/<topic>.json`(`KakomonQuestion[]` origin:'original'、全問に安定 `id` 必須)。

### 論点ページ規格(F1')

```
- H1: 「(論点名)の練習問題(全N問)|(科目名)」
- 論点解説(300字以上): この論点で問われる知識の要点・制度根拠 ← ページの独自価値の核
- 問題1〜N: 回答型UI(下記)
  - 各問: 問題文 / 選択肢5 / 正解 / なぜ正解か(制度根拠つき) / 各誤答がなぜ違うか / 覚え方
- 正答率表示(自サイト集計 — §6-2 のアーキテクチャ。データが溜まるまで非表示)← 誰にもコピーできない独自データ
- 監修者ボックス(氏名・資格・監修日 — 科目適性のある監修者名を科目別に表示)
- 次の論点へ / 科目に戻る / 関連記事 / 弱CTA(講座資料請求 or LINE)
```

- **回答インタラクションは「開示型(タップで解答表示)」ではなく「回答型」とする**: 選択肢をタップ→即時に正誤表示→解説展開。正誤イベントが発生しないと正答率集計(§6-2)・弱点ダッシュボード(§6-5)が原理的に成立しないため。正解・解説は初期HTMLに含めCSS/`<details>`で制御(V1 05章 §4 を継承・上書き)し、**解説の開閉はレイアウトシフトを生まない実装(CLS予算<0.1)**とする。1ページ5〜10問の縦長はモバイルで問題ごとのアンカー+進捗表示で緩和。
- **定量下限(品質CI: V1 ツール#6 に追加)**: 論点解説300字以上 / 1問あたり解説合計200字以上 / 誤答理由は全選択肢分 / 制度・法令名の明記1つ以上 / 監修者ID必須 / 全問に `id`。下限を割るページは公開しない。

## 5. 制作パイプライン(drill-batch — ai-company への追加。24章参照)

| 工程 | 担当 | 内容 |
| --- | --- | --- |
| 1. 論点選定 | 人間(月次) | 論点マスタから当月分を選定(需要実測とドリル進捗で優先度)。**月15論点前後を上限**(23章 時間予算) |
| 2. 作問(クリーンルーム) | AI | 論点の知識内容(法令・公的資料)だけを入力に、問題+選択肢+解説を生成。**過去問・市販問題集をプロンプトに入れることを機械的に禁止**(パイプラインが受け付けない) |
| 3. ファクトチェック | AI+人間 | 正解の制度根拠を一次資料で確認(現行制度か・改定予定はないか) |
| 4. 類似性チェック | **機械+人間** | ①**機械的類似度チェック(前段)**: 試験センターが公式公表している過去問本文+市販問題集テキストを照合用DBとして内部保持し(**対照・解析目的の複製は著作権法30条の4〔非享受利用〕の範囲で適法**)、n-gram・埋め込み類似度でスクリーニング。②**高類似の問題のみ人間+監修者が目視突合**(全問目視は月100問超では非現実的なため機械前段を必須とする)。偶然の酷似を検出したら問題を差し替え(旧IDは欠番)。`similarityCheckedAt` をデータに記録 |
| 5. 監修 | 監修者(**科目別体制**) | 全問。正誤・選択肢の紛らわしさ・制度改定の見落とし。**基本2名(介護福祉士・ケアマネ)+医療系科目(「医療的ケア」「こころとからだのしくみ」等)は看護師等の有資格者をスポット追加**し、適性外監修の「監修済み」表示をしない。監修者名は科目別に表示。契約には①1問単価制(H-32: 300〜800円/問)②月次上限問数③離脱時の引継ぎ条項を明記(1名離脱でドリルが全停止しない体制) |
| 6. 公開 | 人間 | 論点単位で逐次公開。公開ゲート(インデックス率90%)は V1 04章 §3 を継承 |

- 目安工数: 1論点(5〜10問)= **合計60〜90分**(AI生成15分+検証・類似チェック・監修往復45〜75分)【H-33】。**月15論点前後 → 13科目・300論点を16〜20ヶ月で完成**(23章 Phase 3〜4)。

## 6. 「Googleに評価される」ための設計(SEO・E-E-A-T)

1. **検索意図の正直さ**: ドリルページのタイトル・メタに「過去問」を使わない(「過去問」で集客して別物を出すのは品質評価上も倫理上も悪手)。狙うのは「問題」「一問一答」「練習問題」「予想問題」「対策」。
   - **例外(情報型クエリ)**: 「過去問の使い方」「何年分やるべきか」「過去問アプリ・サイト比較」等の**情報型・比較型クエリは、提供物と意図が一致するためタイトルでの「過去問」使用が正当**(25章 #111〜113)。これらのページを `/shiken/kaigofukushishi/` 配下の「過去問情報ハブ」とし、ドリルへの内部導線にする。
   - 「解答速報」はタイトルに使わず「合格点・ボーダー予想」に統一(22章 R-11(a))。
2. **独自データの蓄積**: 正答率・よく間違う選択肢の集計は、実過去問サイトが持たない一次データ。「受験者の◯%がこの論点でつまずく」は引用・言及されうる素材(R-09 の被リンク施策と連動)。
   - **正答率収集アーキテクチャ(実DBなし方針と両立)**: 回答タップ時に GA4 カスタムイベント(`drill_answer`: topicSlug × questionId × choiceIndex × 正誤)を送信 → **月次で集計をエクスポートし `data/drill/answer-stats.json` にコミット → ビルド時に静的表示**(再ビルドは月次)。リアルタイム集計・ユーザー別サーバ保存が必要になった時点を V1 06章 §5 の SQLite/KV 移行トリガーに追加する。
3. **鮮度シグナル**: 制度改定のたびに影響論点を更新カレンダー(V1 06章 §4)で洗い出し、`updatedAt`+改定注記を更新。**介護報酬改定は原則3年毎、処遇改善関連は期中改定・補助金が随時**(初稿の「毎年度改定」は誤りのため修正)。旧制度のまま放置された競合問題集との差別化点。
4. **構造化データ**: Quiz/Question マークアップは V1 05章 §2 の方針を継承(表示は期待しない・低コスト範囲のみ)。
5. **回遊とリピート**: 弱点ダッシュボード(localStorage・ログイン不要)+「次の論点」動線+LINE「毎日3問」配信(H-35)で、指名検索リピーターの不在を製品体験で補う。
   - **localStorage スキーマ**: キーは `drill:v1:<questionId>`(値: 直近の正誤・回答日時・回数)。スキーマバージョンをキーに含め、問題差し替え(欠番)時の孤児レコードは読み捨てる。**iOS Safari の7日エビクション等によりデータは端末限りで消える場合がある**旨をダッシュボードに明示(アカウント機能は持たない)。
   - **「毎日3問」LINE配信は既存問題の自動ローテーション**として設計し、配信企画の手作業をゼロにする(選定ロジック: 公開済み論点から順繰り+試験期は頻出論点優先)。

## 7. CV接続

- 学習中ユーザーへの転職CTAはノイズ(V1 07章の判断を継承)。一次CTA = 講座資料請求(実務者研修比較=受験資格に直結)/ 二次 = LINE(毎日3問+合格後キャリア情報)。
- 修了試験対策(初任者・実務者)だけは講座選び直前の層なので資料請求CTAを標準配置。
- 合格発表(3月)後のLINE配信で転職導線へ — V1 の時間差モデルを縮小継続(H-35)。**ASP規約上のLINE配信内アフィリンク可否は案件ごとに要確認(Phase 2 の提携時マイルストーン)。不可の場合は LINE→サイト記事→`/go/` の2段構成を標準とする**(V1 07章 §5 の未解決事項を昇格)。
