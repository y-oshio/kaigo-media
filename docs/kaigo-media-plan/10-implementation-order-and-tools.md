## 2. Fable 5 で優先的に作るべき自動化ツール(優先順)

1人運営の実質的なボトルネックは「制作スループット」と「更新の抜け漏れ」。Fable 5(Claude Code)で作る社内ツールを優先度順に定義する。いずれも新サイトリポジトリ内の `tools/` に置く想定。

| 優先 | ツール | 何を自動化するか | 人間に残す判断 |
| --- | --- | --- | --- |
| #1 | **過去問データパイプライン** | 試験センター公表PDF(問題・正答)→ `KakomonQuestion[]` JSON への構造化、解説ドラフト一括生成、`status` 管理(draft→fact-checked→reviewed) | 正解番号の公式正答表との全数突合、監修依頼、公開判断 |
| #2 | **記事制作ワークフロー** | キーワード→SERP分析→構成案→ガイドライン準拠ドラフト→`【要出典】`抽出→出典候補の提示→校閲チェック(禁止表現・PR表記・frontmatter必須項目) | 独自価値の注入、出典の真偽判断、公開判断 |
| #3 | **内部リンク監査** | 全ページのリンクグラフを解析し、孤立ページ・3クリック超ページ・アンカーテキスト規則違反を検出しレポート | 修正の実施 |
| #4 | **GSCレポート分析** | Search Console API から月次で「11〜30位クエリ」「CTR異常」「インデックス未登録」を抽出し、リライト候補リストを生成 | リライトの優先順位決定 |
| #5 | **e-Stat統計取り込み** | 賃金構造基本統計調査等のe-Stat API/CSV → `SalaryStat[]` JSON 変換、前年値との差分レポート、影響ページ一覧 | 定義変更の有無確認、公開判断 |
| #6 | **品質CI** | ビルド時に: 構造化データ検証 / 監修必須クラスタの `supervisorId` 空チェック / 出典に `checkedAt` 欠落チェック / Lighthouse予算回帰 | — (機械チェックのみ) |
| #7 | **更新カレンダー・リマインダー** | `update-calendar.ts`(統計公表・試験日程・制度改定)を読み、期限が近い更新タスクを毎週通知 | 更新作業そのもの |
| #8 | **カニバリ検知** | 新記事の `targetQueries` を既存全記事と突合し、重複時に警告 | 統合/分割の判断 |

- 作る順番の根拠: #1・#2 は制作スループット(H-08/H-09)を直接決める。#6 は品質の下限を機械的に保証し、R-03(品質濫用)の防波堤になる。この3つは §1 の実装表に工数込みで組み込み済み(Phase 1)。
- **#3・#4・#5・#7・#8 の着手条件(Phase 1〜2 の負荷から外す)**: #3(内部リンク監査)と #8(カニバリ検知)は「公開ページ100超 or 手動確認が回らなくなった時点」、#4(GSC分析)は「リライト運用を開始する6ヶ月目」、#5(e-Stat取り込み)は「47都道府県ページに着手する Phase 3 冒頭」、#7(更新リマインダー)は「最初の統計更新期(2027年3〜4月)の前」。それまでは手動+チェックリストで代替する。
- いずれも「AIが下書き・検出、人間が判断」の分担を崩さない([04章 §3](./04-content-strategy.md) の原則)。
