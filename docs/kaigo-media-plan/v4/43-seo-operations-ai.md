# 43章 SEO運営AI 設計 — 公開後にサイトを「育てる」10 Agent

- 作成日: 2026-07-05
- 位置づけ: 既存 ai-company(記事を**作る**編集部)に対する、公開後にサイトを**育てる**運営部門。
  組織全体への統合は 46章(ai-company V2)参照
- 稼働開始条件: デプロイ+GSC登録+初回index発生後(44章 M1 以降)。それまでは本章は設計のみ

## §1 共通アーキテクチャ

### データ基盤 `data/ops/`(新設・KBの計測データ版)

全Agentは**実測エクスポートファイルのみ**を入力とする(F-1 の計測データ拡張 — 42章 §6-4)。

```
data/ops/
  gsc/           # GSC エクスポート(検索パフォーマンスCSV・URL検査結果)。ファイル名に取得日
  ga4/           # GA4 エクスポート
  serp/          # 手動SERP観測記録(YYYY-MM-DD-<query>.md — 観測日・上位10・SERP機能)
  kpi/           # 月次KPIスナップショット(kpi-YYYY-MM.json)
  decisions/     # 各Agentの提案と人間の採否記録(追記のみ・消さない)
```

- 取得は当面**人間の手動エクスポート**(GSC API 自動化は Phase 2 の任意課題)。
  Agentはデータが無ければ「取得依頼」を出して停止する。数値の推定・補完は禁止
- 全Agentの出力は「提案」であり、実行(リライト・構成変更・公開)は編集部パイプライン+人間承認を経る

### 共通禁止事項(全Agent)

- 計測値の捏造・外挿・「体感」数値化 / KBにない事実の記事への追加提案
- 検索順位操作目的の隠しテキスト・誘導ページ・AI生成の薄いページ量産等、
  Google スパムポリシー抵触施策の提案
- 競合サイトのコンテンツ複製・言い換え転載の提案
- ゲート値(noindex条件等)の変更提案を「実行」すること(提案は可・変更は高性能モデル+人間)

### 実行サイクル(定常運転)

| 頻度 | 実行するAgent |
| --- | --- |
| 週次 | GSC Analyzer → Index Coverage Monitor → KPI Manager(短報) |
| 隔週 | CTR Optimizer / Internal Link Optimizer |
| 月次 | Keyword Opportunity Finder / Content Decay Detector / Rewrite Planner / KPI Manager(月報) |
| 四半期 | SERP Watcher(定点観測)/ Backlink・Digital PR Planner |

## §2 Agent定義

### A1. GSC Analyzer(検索パフォーマンス分析)

- **役割**: GSCデータから「何が起きているか」の事実サマリーを作る。全運営Agentの上流
- **入力**: `data/ops/gsc/` の検索パフォーマンスCSV(直近28日+前期比較)
- **出力**: `data/ops/kpi/` へ週次サマリーmd(クエリ×URL×クリック/表示/CTR/順位の変動上位、
  新規流入クエリ、急落クエリ)
- **使用データ**: GSCエクスポートのみ
- **実行頻度**: 週次
- **判断基準**: 変動の報告閾値=クリック±30%または表示±50%(母数10未満のクエリはノイズとして除外)
- **禁止事項**: データに無い期間への言及、順位の将来予測を断定で書くこと
- **人間承認**: 不要(事実サマリーのみ。施策提案はしない)
- **軽量モデル**: **可**(集計と閾値判定のみ)

### A2. Keyword Opportunity Finder(獲得機会発見)

- **役割**: 「順位11〜30位で表示が多いクエリ」「KBで書けるのに記事がないクエリ」を発見し、
  新規記事・追記の候補リストを作る
- **入力**: A1サマリー、KBカタログ(data/kb/)、既存記事一覧(content/)、02章キーワード戦略
- **出力**: 機会リスト(クエリ / 現状順位 / 対応=新規記事 or 既存追記 / 必要なKBデータセット /
  F-1充足可否)→ `data/ops/decisions/` に提案として追記
- **使用データ**: GSC実測+KB+既存記事
- **実行頻度**: 月次
- **判断基準**: 優先度 = 表示回数 × (11〜30位) × KB充足度。**KBに事実が無いテーマは
  「KB-2投入待ち」と分類し、記事化候補に入れない**(F-1)
- **禁止事項**: 検索ボリュームの推定値を実測のように書く/YMYL断定が必要なテーマの提案
- **人間承認**: 記事化着手の採否(編集部パイプラインへの投入判断)
- **軽量モデル**: **可**

### A3. CTR Optimizer(タイトル・description改善)

- **役割**: 順位のわりにCTRが低いURLを特定し、title/description の改善案を出す
- **入力**: A1サマリー(URL別・順位帯別CTR)、対象記事の現frontmatter
- **出力**: URLごとに現行/改善案/根拠(どの実測値か)の3点セット、最大10件/回
- **使用データ**: GSC実測+記事frontmatter
- **実行頻度**: 隔週
- **判断基準**: 対象=順位1〜10位かつ表示100以上で、CTRが同順位帯の自サイト中央値の1/2未満。
  変更は1URLにつき1要素ずつ(効果測定可能性の確保)。再変更は最低4週間空ける
- **禁止事項**: 誇張・釣りタイトル(内容が担保しない約束)、統計値のタイトル使用時にKB値と不一致
- **人間承認**: 全変更(frontmatter変更=公開物変更のため)
- **軽量モデル**: **可**

### A4. Content Decay Detector(劣化検知)

- **役割**: クリック・順位が持続的に下がっている記事、および**情報鮮度が切れた**記事
  (KB更新カレンダーで新年度統計が出たのに旧値のままの記事)を検出する
- **入力**: A1サマリーの時系列(8週以上)、`data/update-calendar.ts`、記事のsources checkedAt
- **出力**: 劣化リスト(URL / 劣化種別=順位劣化 or 鮮度切れ / 深刻度)→ A9 Rewrite Planner へ
- **使用データ**: GSC実測+更新カレンダー+frontmatter
- **実行頻度**: 月次
- **判断基準**: 順位劣化=4週連続で順位悪化かつクリック−30%。鮮度切れ=参照統計に新年度版が
  公表済み(カレンダー起点)or checkedAt から12ヶ月超
- **禁止事項**: 一時変動(SERP全体の変動)と個別劣化の混同 — 全サイト一斉下落時はA6へ回す
- **人間承認**: 不要(検出のみ。対処はA9経由で承認)
- **軽量モデル**: **可**

### A5. Internal Link Optimizer(内部リンク最適化)

- **役割**: 孤立記事・リンク過少ページを検出し、文脈に合う内部リンク追加案を出す。
  特に「47県ページ⇔給料ガイド」「記事⇔ツール・診断」の層間リンクを監視
- **入力**: content/ 全記事の本文リンク、routes構成、A1のURL別流入
- **出力**: 追加リンク案(リンク元URL / 挿入位置の文脈 / リンク先 / アンカーテキスト案)
- **使用データ**: リポジトリ内実データ(content/・pages/)+GSC実測
- **実行頻度**: 隔週(記事公開が続く期間)/ 月次(定常期)
- **判断基準**: 全公開記事は最低2本の内部リンクを受ける。アンカーテキストは自然文
  (キーワード完全一致の連発禁止)。1記事への追加は1回3リンクまで
- **禁止事項**: 文脈と無関係なリンク挿入、/go/(アフィリエイト)リンクの「内部リンク」扱い
- **人間承認**: 本文変更を伴うため全件(編集部パイプライン経由)
- **軽量モデル**: **可**

### A6. SERP Watcher(検索結果環境の定点観測)

- **役割**: 主要クエリのSERP構成(上位ドメイン・SERP機能・AI Overviews の有無)を定点記録し、
  戦略前提(T-0c仮説・H系仮説)の変化を検知する
- **入力**: `data/ops/serp/` の観測記録(人間またはスクリプトによる手動観測)
- **出力**: 前回比の変化レポート(新規参入ドメイン / SERP機能の増減 / 自サイト面の変化)。
  戦略仮説に反する変化は「高性能モデル案件」としてフラグ
- **使用データ**: 手動SERP観測記録のみ(スクレイピング自動化はGoogle規約リスクがあるため、
  観測は人間の目視記録を正とする)
- **実行頻度**: 四半期(P8着手前はT-0cとして臨時実行)
- **判断基準**: 定点クエリ20個(給料系10・ドリル系5・診断系5)を固定。クエリの入れ替えは人間承認
- **禁止事項**: 未観測クエリのSERP構成を記憶や推測で書くこと(観測日つき記録のみが根拠)
- **人間承認**: 定点クエリの変更、戦略変更を伴う結論
- **軽量モデル**: **可**(差分検出のみ。戦略転換の判断は高性能モデルへ)

### A7. Backlink / Digital PR Planner(被リンク・デジタルPR企画)

- **役割**: 引用されやすい一次加工データ(都道府県別給料統計等)を核に、被リンク獲得企画を立てる。
  マモリビの構造的強み=「出典明記の統計データ」を外部に届ける唯一のAgent
- **入力**: KB(C2ほか)、公開済み統計ページ一覧、A1の被リンク経由流入(参考)
- **出力**: PR企画案(データの切り口 / 想定引用者=記者・自治体・介護系ブログ / 配布形式 /
  プレスリリース草稿)。**送付・公開は全て人間**
- **使用データ**: KB verified/published のみ
- **実行頻度**: 四半期
- **判断基準**: 企画は「KBに実在する数値だけで完結する」こと。誇張見出し(「最下位は◯◯県!」の
  煽り単独)より文脈提供(全国比較+出典)を優先
- **禁止事項**: リンク購入・相互リンク集・PBN等の提案(Google リンクスパム抵触)、
  データの切り口操作による誤解誘導(軸の省略等)
- **人間承認**: 全企画の実行・全外部送付
- **軽量モデル**: 企画の**下書きは可**。切り口の妥当性判断(誤解誘導になっていないか)は
  人間+必要に応じ高性能モデル

### A8. KPI Manager(KPI管理)

- **役割**: 08章KPIロードマップ+44章マイルストーンに対する現在地を毎月確定させ、
  乖離があれば原因仮説と対処候補を提示する
- **入力**: `data/ops/kpi/` スナップショット、A1〜A7の全出力、44章の目標値
- **出力**: 月報(`data/ops/kpi/report-YYYY-MM.md`): 実績 vs 目標 / 乖離要因 / 来月の重点 /
  高性能モデル・人間へのエスカレーション事項
- **使用データ**: 実測KPIのみ
- **実行頻度**: 月次(+週次短報)
- **判断基準**: マイルストーン到達判定は44章の定義に従う。2ヶ月連続で計画比−50%なら
  「戦略見直し(高性能モデル案件)」を自動フラグ
- **禁止事項**: 未達の粉飾・楽観バイアス(「来月には改善見込み」を根拠なく書く)
- **人間承認**: エスカレーション後の意思決定
- **軽量モデル**: **可**

### A9. Rewrite Planner(リライト計画)

- **役割**: A3/A4/A5の検出結果を束ね、「どの記事を・何を根拠に・どう直すか」の月次リライト
  キューを作る。編集部パイプラインへの発注書を書くAgent
- **入力**: A3のCTR案・A4の劣化リスト・A5のリンク案、KB更新差分(新年度統計等)
- **出力**: リライト指示書(URL / 変更種別 / 根拠データ / 変更範囲 / 検収観点)最大5本/月
- **使用データ**: 上流Agent出力+KB
- **実行頻度**: 月次
- **判断基準**: 優先度 = 流入インパクト × 修正コストの逆数。**鮮度切れ(統計旧値)は常に最優先**
  (誠実性はトラフィックより優先)。同一記事のリライトは8週間隔
- **禁止事項**: リライトを装った水増し(文字数目的の加筆)、KB未登録の新事実追加
- **人間承認**: キューの承認(実行前)+リライト後の検収(42章 §5)
- **軽量モデル**: **可**

### A10. Index Coverage Monitor(インデックス被覆監視)

- **役割**: 「公開したのにindexされない」「noindexのはずがindexされている」を検知する。
  公開ゲート(sitemap/noindex)の実世界での整合性監視
- **入力**: `data/ops/gsc/` のページインデックス登録レポート、sitemap.xml の実出力、
  各ページの意図状態(公開ゲート定義から導出)
- **出力**: 不整合リスト(URL / 意図=index・noindex / GSC実態 / 推定原因)。
  ゲート実装の矛盾を示唆する場合は「高性能モデル案件」フラグ(41章 §3 の再発検知)
- **使用データ**: GSCインデックスレポート+リポジトリのゲート定義
- **実行頻度**: 週次(公開直後期)→ 月次(安定期)
- **判断基準**: 意図と実態の全件突合。「クロール済み未登録」が4週続くURLは品質シグナルとしてA9へ
- **禁止事項**: 不整合の自動修正(原因分析なしのsitemap手修正等)
- **人間承認**: 修正方針の承認
- **軽量モデル**: **可**(突合のみ。原因がゲート実装にある場合は高性能モデルへ)

## §3 立ち上げ順序

1. **M1(初index後)**: A1・A10 のみ稼働(データが薄い時期に分析Agentを増やしても無意味)
2. **M2〜M3(記事量産期)**: A5・A8 追加
3. **表示1万/月 到達後**: A2・A3・A4・A9 追加(統計的に意味のある母数が出てから)
4. **Phase 2以降**: A6・A7 追加

GSC分析ワークフロー(人間側): 週1回 GSC から検索パフォーマンスCSV+インデックスレポートを
エクスポート → `data/ops/gsc/` に保存 → Claude Code で A1→A10 を実行 → 提案を採否 →
`data/ops/decisions/` に記録。所要15〜30分/週を想定。
